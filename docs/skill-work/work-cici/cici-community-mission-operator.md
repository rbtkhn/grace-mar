## Project role of `cici-ai`

`cici-ai` is the lightweight social front door for Cici's OB1 / companion-self learning cohort.

## Mission statement

`cici-ai` helps AI-curious builders learn by doing: each member is invited to set up their own governed OB1 / Cici-style workspace, ask for help without shame, share visible progress, and gradually become capable of helping the next person. The team exists to turn private curiosity into durable practice - safe setup, real artifacts, repeatable habits, and peer support - while keeping every person's Record, repo, and infrastructure under their own control.

Its job in the work-cici project is to convert interest into visible initiation:

- gather a small team of AI-curious hobbyists around Cici's learning path
- help members fork or stand up their own OB1 / Cici-style workspace safely
- normalize questions, stuck states, and screenshot-based troubleshooting in public
- reduce repeated one-on-one advisor load by making the golden path social and repeatable
- produce evidence-linked activation signals for the advisor lane: join, goal stated, fork/opened workspace, first task, return within seven days, artifact, and helper behavior

What `cici-ai` is not:

- not Cici's Record and not a place to merge identity truth
- not the canonical Cici repo; GitHub remains the durable technical spine
- not a passive announcement channel where joins alone count as success
- not BrewMind partner-facing marketing copy
- not a paid program in this lane

 The Telegram team is therefore best understood as a **cohort activation surface**: chat supplies warmth, momentum, and help-seeking; GitHub supplies durable artifacts; work-cici supplies advisor evidence, measurement, and coaching.

## Primary risk (operator choice): member drop-off

Drop-off is treated as the **main** failure mode (versus only advisor time or Xavier overload). It is driven less by missing README detail than by **shame**, **silent stalls** (Git, Supabase, first session), and **no felt “win”** before life interrupts.

### What usually kills the cohort

- **Fuzzy “done”** — “In the team” feels like progress; real milestone is **fork + own Supabase + no secrets leaked** (adjust labels to match pinned rules).
- **Silent failure** — Stuck users **don’t post** and vanish.
- **Long path to first dopamine** — If session one does not end with **something working** (even small), motivation dies.
- **Async shame** — Mitigated by **blame-free** norms and **modeling** questions in public (see Neural Sandbox rules).

### High-leverage mitigations (low meeting load)

1. **One shouted milestone** — Same definition in pinned message, checklist, and welcome: what **counts as set up**.
2. **Checkpoints** — e.g. Day 2 and Day 7: “Reply **done** or **stuck on X**” — **stuck** replies are the leads that matter.
3. **Smallest first win** — Order steps so **GitHub + fork** can ship **before** heavier MCP/Supabase if needed; or pair early steps with a **5-minute win**.
4. **Name silence** — “If you have not posted 48h after joining, you are probably stuck — **paste the error** (screenshot ok).”
5. **Structure beats scatter** — Ten solos drop more than **two waves of five** or **accountability pairs** with the same curriculum.

### Funnel metrics (honest)

Track **stages**, not only headcount:

| Stage | Example signal |
|-------|----------------|
| Joined team | In Telegram (**cici-ai** invite used) |
| Initiation **done** (counts toward 10) | **OB1 / Cici instance** path complete per published checklist (fork + own stack + no leaked secrets) |
| **7-day active** | At least one check-in or visible artifact per **rolling week** (define with Cici) after initiation |
| GitHub account | Username posted (see transcript pattern) |
| Fork | Link to **their** fork |
| Supabase | Project created (self-report ok) |
| First working session | Define minimally (e.g. Claude Code opens repo, or MCP ping) |

Measure **where the stairs get steep**, not vanity joins.

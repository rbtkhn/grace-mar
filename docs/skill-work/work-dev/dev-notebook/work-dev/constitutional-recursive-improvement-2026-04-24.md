# Grace-Mar: Constitutional Recursive Improvement

**Report dated:** 2026-04-24  
**Territory:** WORK / dev-notebook architecture-positioning note — **not** SELF, **not** EVIDENCE, **not** Voice knowledge.  
**Scope:** Reusable framing for architecture and positioning work. This note summarizes the repo's live improvement structure as of the report date; it does not create new canonical doctrine beyond the linked source surfaces.

## Core claim

Grace-Mar's core innovation is **constitutional recursive improvement**.

The system is designed so intelligence can compound across sessions, runtime observations, strategy work, dev tooling, receipts, expert threads, and derived artifacts **without allowing that compounding to silently rewrite canonical truth**.

That is the important distinction. Grace-Mar is not primarily unusual because it has memory, evals, traces, notebooks, dashboards, or self-improvement loops. Many systems now have those. What is rarer is the governing structure that keeps improvement from collapsing into self-corruption:

```text
runtime observation
-> bounded context or proposal material
-> contradiction / review / candidate staging
-> human approval or rejection
-> sovereign merge into canonical Record only through the authorized path
-> future behavior inherits the improved state or improved workflow
```

The constitutional part matters as much as the recursive part:

- the durable Record remains the four governed surfaces in [`docs/runtime-vs-record.md`](../../../../runtime-vs-record.md)
- WORK remains WORK, not identity truth
- derived artifacts remain **derived / rebuildable / non-canonical**
- contradiction is preserved before synthesis
- external runtimes and helpers may stage, inspect, and propose, but they do not hold merge authority

In short, Grace-Mar improves recursively under explicit memory boundaries rather than through undifferentiated "the AI remembers" behavior.

## Top distinctive loops

These are the loops where Grace-Mar feels most differentiated from ordinary agent-memory or AI-dev systems.

1. **Contradiction / identity-diff loop**  
   New evidence does not simply overwrite the active model. It is compared against the current state and surfaced as a governed identity-change question with explicit before / proposed / after logic. See [`docs/CONTRADICTION-ENGINE-SPEC.md`](../../../../CONTRADICTION-ENGINE-SPEC.md).

2. **Three compounding loops separation**  
   Grace-Mar explicitly separates **Record**, **WORK**, and **Repo / CI** as different compounding loops. This prevents drafts, runtime convenience, or green CI from being mistaken for canonical truth. See [`docs/skill-work/work-dev/three-compounding-loops.md`](../../three-compounding-loops.md).

3. **Canonical Record gated-growth loop**  
   Durable identity change happens through signal detection, candidate staging, approval, and the authorized merge path. This is stronger than ordinary chat-memory personalization because the system treats merge authority as sovereign rather than ambient. See [`docs/runtime-vs-record.md`](../../../../runtime-vs-record.md).

4. **Conflict-preservation loop**  
   Unresolved contradiction is preserved as a governed object instead of being flattened away for narrative smoothness. Growth, refinement, contradiction, and replacement are distinct relations, not one generic update action.

5. **Strategy promotion / demotion ladder**  
   Material rises because it becomes more reusable, testable, or decision-relevant, not because it merely exists or sounds persuasive. Weak material may also be demoted. That gives strategic cognition an anti-rhetoric discipline.

6. **Tacit capture loop**  
   Tacit operator knowledge can be captured, normalized, and prepared for review without automatically becoming memory authority. This makes lived operator knowledge legible while preserving the gate.

7. **Runtime observations -> prepared context loop**  
   Runtime notes improve future work through bounded context-building while remaining explicitly non-canonical. Observation can support better action without silently turning into truth.

8. **Memory brief -> gate candidate bridge**  
   Runtime memory can inform a candidate, but it cannot become durable truth directly. The bridge into canon is explicit and governed rather than automatic.

9. **Dream / Coffee bounded continuity loop**  
   Continuity is maintained through explicit rituals that support consolidation and re-entry without turning maintenance into autonomous self-editing. This is continuity with boundaries, not autonomous memory absorption.

10. **Runtime complements membrane loop**  
   External runtimes may return observations, exports, imports, or candidates, but they do not become sovereign memory. The membrane preserves experimentation without surrendering canonical control. See [`docs/runtime-vs-record.md`](../../../../runtime-vs-record.md) and runtime-complements docs referenced there.

## Known-pattern loops, Grace-Mar-customized

These loops are still important, but their value comes less from originality and more from how Grace-Mar integrates them under a governance-aware memory doctrine.

1. **Observability / receipts / replay**  
   Tracing, replay, and audit surfaces are now standard serious AI engineering practice. Grace-Mar's distinctive move is to keep them repo-native and explicitly subordinate to governed Record truth rather than letting observability stand in for truth.

2. **Agent reliability eval flywheel**  
   Turning failures into eval cases and regression checks is common in mature AI engineering. Grace-Mar gains leverage by tying this to bounded autonomy, capability claims, and governance constraints rather than treating evals as a free-standing optimization game.

3. **Workbench artifact iteration**  
   Generate -> run -> inspect -> revise -> receipt is a familiar engineering loop. Grace-Mar's contribution is the boundary language: a successful artifact run is not the same thing as external truth or canonical memory. See [`docs/skill-work/work-dev/workbench/README.md`](../../workbench/README.md).

4. **Forecasting / backtest loop**  
   Forecast -> compare to actuals -> improve is an established pattern. Grace-Mar customizes it by keeping forecasts inside WORK unless a human separately stages a downstream conclusion for review. See [`docs/runtime-vs-record.md`](../../../../runtime-vs-record.md).

5. **Run receipts / session envelopes**  
   Structured run state and resumable session traces are standard engineering patterns. Grace-Mar applies them to strategy and operator work in a way that improves legibility without granting merge authority.

6. **Claim-proof / capability status loop**  
   Backing claims with tests, scripts, receipts, or demos is ordinary software-quality discipline. Grace-Mar adapts it to a companion-governance context, where capability claims must not outrun actual proof.

7. **Derived regeneration loop**  
   Rebuild manifests, health summaries, and per-artifact rationale sidecars are not unprecedented, but Grace-Mar now uses them to make derived outputs inspectable without blurring them into canonical surfaces. See [`artifacts/README.md`](../../../../../artifacts/README.md) and [`docs/skill-work/work-dev/derived-regeneration.md`](../../derived-regeneration.md).

8. **Dashboards and review surfaces**  
   Boards, dashboards, and derived summaries are common support layers. Grace-Mar's contribution is that these surfaces remain explicitly subordinate to canonical files and governed review paths rather than being quietly promoted into authority.

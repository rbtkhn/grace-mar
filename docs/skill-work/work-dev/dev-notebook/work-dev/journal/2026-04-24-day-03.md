## What clicked

- **Naming is architecture.** The session kept proving that the hardest part was not writing another script; it was refusing terminology that would quietly blur policy boundaries. “Shadow” works as a metaphor; it becomes dangerous when it starts behaving like a replacement ontology.
- **Rebuildability gets stronger when it explains itself.** The sidecar idea matters because it lets a derived file carry its own rebuild provenance without pretending to be canonical truth, portable-record rationale, or a merge receipt.
- **System maps have to be implementation-faithful to be useful under pressure.** A map that is 85% right but misses the live loops trains the wrong instincts when you are designing the next wedge.

## Friction / questions

- The rebuild flow is now coherent for the covered targets, but the repo still ended the session with **two transient rebuild receipts** and an **untracked strategy-notebook graph sidecar**. That is a useful reminder that tracked-vs-transient policy still needs one more pass at the edges.
- The recursive-improvement map is good at the level of doctrine, but if it is going to become a repo doc it should say less about abstract “approved state” and more about the actual surfaces that now exist.

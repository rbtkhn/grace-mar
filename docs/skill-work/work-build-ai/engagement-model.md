# work-build-ai engagement model

How the `work-build-ai` business layer should package work commercially.

The main rule:

- trust and diagnosis come before retainers
- bounded implementation comes before large platform promises
- sell clarity first, then build

---

## Engagement modes

| Mode | When to use | What is included | Notes |
|------|-------------|------------------|-------|
| **Paid diagnostic** | First engagement | Current-state review, risk map, and best-next-step memo | Default starting point. |
| **Architecture pass** | After the diagnostic when the buyer wants a clearer contract | `record` vs `runtime` separation, export / handback design, approval boundaries | Best for teams with multiple runtimes or memory sprawl. |
| **Fixed-scope implementation sprint** | After the architecture is clear | One bounded workflow or integration surface | Do not stack too many surfaces into the first sprint. |
| **Advisory retainer** | Only after clear trust and repeated need exist | Ongoing review, prioritization, and iterative guidance | Should be earned, not forced early. |

---

## Commercial sequencing

1. Diagnostic
2. Architecture pass if needed
3. One bounded implementation sprint
4. Advisory or repeated sprint work only if the relationship justifies it

---

## Guardrail

Do not jump straight from first contact to a broad retainer unless the buyer already understands the pain and the value of a governed architecture approach.

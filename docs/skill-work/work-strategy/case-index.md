## 1. What this file is

This file is the local operator-facing index of historical and civilizational cases that are repeatedly useful in work-strategy.

It is not a mirror of civilization_memory. It is a thin working index that helps the operator move from live events to reusable mechanisms, fit checks, mismatch checks, and better strategic judgment.

This file should stay concise, searchable, and operational.

## 2. What this file is not

This file is not:

- a full historical archive
- a replacement for civilization_memory
- a replacement for the History Notebook
- a place for long essays
- a place to store identity claims
- a place to turn analogy into certainty

If a case needs long treatment, put that in the History Notebook or upstream materials and keep only the thin operator-useful abstraction here.

## 3. Boundary rules

### 3.1 WORK-only
Everything here is WORK-only unless separately promoted through the proper review path.

### 3.2 No automatic Record effects
Nothing here updates SELF, EVIDENCE, or any identity-facing surface.

### 3.3 Thin-entry discipline
Each case entry should stay short enough to be useful during live analysis.

### 3.4 Mechanism-first
Each case must be framed around reusable mechanism, not just chronology.

### 3.5 Falsifier required
A case without mismatch logic or a falsifier is not ready for repeated use.

## 4. When to add a case

Add a case here only if one or more of the following is true:

- it has already been used multiple times in strategy notes, current-events analysis, or decision points
- it captures a reusable mechanism better than existing cases
- it consistently improves judgment rather than merely sounding intelligent
- it anchors a recurring work-strategy lane
- it reduces analogical drift by giving the operator a cleaner object to cite

If a case is interesting but not yet reusable, keep it in a resonance note or the History Notebook instead.

## 5. How to use this file

The standard workflow that invokes this check is [current-events-analysis.md](current-events-analysis.md) (§5.4 Case-index check). That pipeline requires checking this index before proposing any historical or CIV-MEM parallel.

Typical use flow:

1. A live event, brief, transcript, or question enters work-strategy.
2. Check whether an existing case here matches the mechanism.
3. If a case fits, cite it with fit / mismatch / falsifier notes.
4. If no case fits well, create a resonance note instead of forcing a bad parallel.
5. If the new pattern recurs, add or revise a case entry.
6. If a case begins to need richer local treatment, create or update a History Notebook chapter.

Case families and lenses that seed new cases are defined in [civilizational-strategy-surface.md](civilizational-strategy-surface.md).

## 7. Reading rule

A case should only be cited strongly when the operator can say all three:

- why this fits
- why this may not fit
- what would falsify the parallel

If that standard cannot be met, use the case illustratively or use a resonance note instead.

- Region / system: Global energy order
- Mechanism: Chokepoint coercion and resource leverage producing second-order political and economic effects
- Best use: Energy pressure, external dependency, price shock, alliance strain, inflationary strategic spillover
- Strongest fit signals:
  - concentrated supply leverage
  - abrupt price or access shock

  - diffuse supply base
  - weak chokepoint control

## 9. Compression rule

If this file grows unwieldy, compress by:

- merging duplicative cases
- reducing prose
- moving richer treatment into the History Notebook
- keeping only the operator-useful abstraction here

The point of this file is not completeness. The point is strategic reuse.

## 10. Promotion path

Cases in this index often serve as Stage 1 (case hit) inputs to the [promotion ladder](promotion-ladder.md). When a case citation in the strategy notebook or current-events analysis proves reusable enough to deserve a resonance note, analogy audit, watch support, or doctrine note, the ladder defines the standard path and reasoning standard for that promotion.

## 11. Review protocol

When reviewing or revising entries:

1. Check whether an existing case already covers the mechanism.
2. Prefer sharpening fit / mismatch / falsifier over adding prose.
3. Retire or compress cases that are ornamental, redundant, or rarely used.
4. Promote repeated local use into History Notebook chapters only when richer treatment is justified.
5. Keep labels stable once they become habitual in strategy writing.

## 12. Minimal maintenance standard

This file is healthy when:

- cases are easy to scan
- cases improve judgment
- cases reduce analogical sloppiness
- cases are linked to real strategy use
- cases remain thinner than notebook chapters
- the operator can quickly tell when not to use a case

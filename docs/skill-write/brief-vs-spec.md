# Brief vs spec

WORK only; not Record.

## Brief

A **brief** states intent: what the output must achieve, who it is for, what good looks like, examples, constraints, and the acceptance bar.

Use a brief for writing, strategy synthesis, creative work, public-facing output, and agent delegation where taste or judgment matters.

## Spec

A **spec** states system behavior: interfaces, files, data flow, validation, edge cases, and implementation constraints.

Use a spec for code, schemas, scripts, integrations, automation, and runtime behavior where another engineer or agent must build the same thing without guessing.

## Grace-Mar default

For non-trivial WORK output, write or infer a brief first. If the task turns into system behavior, add a spec or tracer-bullet plan rather than stretching the brief into engineering detail.

Briefs reduce drift; specs reduce ambiguity. Neither is gate approval, Record truth, or MEMORY.
